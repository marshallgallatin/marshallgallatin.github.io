## 2n + 1... no wait

This week we received the first assignment.  Admittedly, I have implemented this particular assignment before, so there isn't anything really suprising.  I'm looking forward to implementing the cached version though, theres so many different things to try and I've been thinking that I can implement it even better than last time.

To be honest, class this week was kind of boring to me.  Monday was introduction to the problem, Wednesday was about mathematical tricks to reduce the cycle count, and Friday was about caches and possible cache implementations.

Working on the project itself was kinda fun.  I've wanted to learn more about using github and this was perfect.  We used github for storing our CS439 projects, but didn't do anything more complicated than pull and push.  In fact, once we tried to revert some changes, and we ended up in crazy branching hell where we ended up making a copy of the reverted files, syncing with the master, and then copying the reverted files into the master.  Now I know how to make a branch and merge the changes into the master, pretty slick.  I've used other revision control software before (Subversion and Perforce come to mind), but git takes a little getting used to.  The main thing that is problematic for me, is that I'm used to thinking about checking out files, changing them, then pushing those files back in. Git seems to be more of a snapshot idea - the whole repository is in one state, changes happen, then the new state is commited to the repository.  I don't know if either philosphy is better, I think they are just different (and like most things, the correct answer is probably *it depends*).

The thing I like least about the project is probably the instructions.  I'm not sure if this is done on purpose or not.  It took several readings to figure out exactly was expected of us.  Once I got started creating the issues, things got much clearer about what steps I needed to take to get everything done (so maybe it is on purpose).  I like how you can close issues via a commit - forgetting to close issues is frequently a problem when the revision control software is separate from the bug tracker.

I think I'm going to play around with changing the theme to this blog.  I *finally* got the `jekyll serve` command to work locally.  Ended up having to install a ruby gem to get it all to work.

For some reason, working on this project makes me want to eat at Schlotzky's.

### Tip of the week ###
Milestones in github are pretty nice.  You can add your issues to them which serves two main purposes:

1. Makes a large task seem smaller, and easier to complete.
2. Gives the ability to organize the issues into logical groups.

After creating issues for the project, I then created milestones.  This helped me break up the tasks into smaller groups which makes the project easier to work on a little bit at a time. I now have 4 milestones with about 4 issues each, which when I finish all 4 milestones, I will be done with the project (Two down, two to go).
